# hannom-app — Deployment & Hosting Guide

A hand-off document for hosting the Han-Nom OCR corpus builder.

**Target scale:** 3–4 concurrent reviewers, each responsible for ~3–600 pages
(so a whole corpus of up to ~2,400 pages, OCR'd once, then reviewed by hand).

**TL;DR recommendation:** one modest Linux VM running `docker compose`, behind a
reverse proxy (Caddy) for HTTPS, with **nightly Postgres backups**. Kubernetes is
overkill at this scale. Everything is already containerized and stateless except
Postgres + a data disk.

---

## 1. Architecture

Three containers + two pieces of state. All defined in `docker-compose.yml`.

```
                          HTTPS (443)
   Reviewers' browsers  ─────────────►  Caddy / nginx  ──►  app  (FastAPI, :8000)
   (login, review UI)                   (TLS, domain)        │  stateless, no GPU
                                                             │
                        ┌────────────────────────────────────┼───────────────┐
                        │                                     │               │
                        ▼                                     ▼               ▼
                  Postgres (:5432)  ◄──── job queue ────►  worker(s)     ./data disk
                  jobs, records,          (claim/run)      PaddleOCR      uploads/,
                  users, assignments                        CPU or GPU    output/pages/
                  [named volume: pgdata]                                  [bind mount]
```

- **app** — FastAPI (uvicorn). Serves the single-page UI, auth, the REST API, and
  serves page images. Stateless, holds **no secrets of its own** and needs **no
  GPU**. Image ≈ 350 MB.
- **worker** — Pulls jobs off the Postgres queue and runs the OCR/extraction
  pipeline (PaddleOCR + PDF rendering via poppler). CPU by default; GPU-capable.
  Image ≈ 1.8 GB. **You can run several** (see §4).
- **postgres** — The source of truth: `jobs`, `records`, `users`, `assignments`.
  Tiny data footprint (records are small text rows; ~8 MB for the sample).
- **State 1 — `pgdata` (named volume):** the database. **Back this up.**
- **State 2 — `./data` (bind mount / disk):** uploaded PDFs (`uploads/`), rendered
  page images (`output/pages/`, ~2 MB/page PNG), JSONL exports, scratch. Back up
  `uploads/` (page images can be regenerated by re-running a job; the DB records
  can't).

The app and worker are **decoupled by the Postgres queue** — a worker crash never
loses a reviewer's edits, and the app stays responsive while OCR runs in the
background.

## 2. End-to-end workflow

**A. Bulk OCR (one-time, background, heavy):**
1. Admin logs in, uploads a Châu Bản PDF → app stores the file under `uploads/` and
   inserts a `pending` job row in Postgres.
2. A worker atomically claims the job (`UPDATE … FOR UPDATE SKIP LOCKED`), renders
   each page at `PDF_DPI` (default 300), OCRs the Hán column, pairs it with the
   Vietnamese text layer, and **bulk-inserts the extracted records into Postgres**.
   Page images are written to `output/pages/` for the review UI.
3. Job marked `done`. This is the slow part (see §3).

**B. Review (interactive, light, the day-to-day):**
4. Admin assigns each reviewer a **page range** on the job (A → 1–200, B → 201–400…).
5. Reviewers log in and see the whole document but can only **edit/verify their
   assigned pages** (others are read-only). They fix Hán/Quốc-ngữ, adjust boxes,
   re-OCR a box, or use **vision-LLM assist** (draw a box → the browser crops it →
   sends the crop + current OCR text to the reviewer's own LLM key → writes back the
   reading). Saving persists to Postgres.
6. Export: `GET /jobs/{id}/output` regenerates JSONL from the DB (reflecting edits).

**Reliability techniques already in the code:**
- **Postgres queue with `SKIP LOCKED`** → multiple workers pull jobs concurrently
  without stepping on each other.
- **Orphan recovery** → on startup a worker requeues any job left `running` by a
  crashed worker (`requeue_running`).
- **Idempotent schema init** → tables are created on startup if missing; safe to
  restart/redeploy.
- **Healthchecks + `restart: unless-stopped`** on all services; app waits for
  Postgres to be `healthy` before starting.
- **Per-user LLM keys** are held in the browser and sent per-request — **never
  stored or logged server-side**. The server needs no LLM keys.

## 3. Sizing for this scale

### Storage
| Item | Per unit | 2,400 pages |
|---|---|---|
| Rendered page PNG (300 DPI) | ~2 MB | ~5 GB |
| Source PDFs (text-layer) | ~0.1–0.2 MB/page | ~0.3–0.5 GB |
| Postgres data (records + meta) | tiny | < 100 MB |
| Docker images | app 0.35 GB + worker 1.8 GB | ~2.2 GB |

→ **Provision ~30 GB disk** (comfortable headroom + growth). Page images dominate;
lowering `PDF_DPI` shrinks them and speeds OCR at some accuracy cost.

### CPU / RAM
- **Interactive review load is trivial** — 3–4 people clicking is a few requests/sec.
- The real cost is the **one-time bulk OCR**, which is CPU-bound.
- **Budget ~10–30 s/page on CPU** (render + detect + recognize; **benchmark on the
  target hardware** — it varies a lot). A GPU (e.g. an NVIDIA card) is ~5–10× faster.
- **Estimate:** 2,400 pages ≈ **7–20 hours on one CPU worker**; run **N workers**
  (one per free core) to divide that, or use a **GPU worker** to finish in ~1–3 h.

### Recommended VM
- **Baseline:** 4 vCPU / 8 GB RAM / 30 GB SSD → runs everything; bulk OCR with 2–3
  CPU workers over a few hours; smooth review afterward.
- **Faster OCR:** 8 vCPU / 16 GB (more parallel CPU workers) **or** add a GPU and set
  `OCR_USE_GPU=1` with a CUDA paddle wheel (see `Dockerfile.worker` notes).
- Each CPU worker uses ~1–2 GB RAM; size RAM to `~2 GB × workers + 2 GB`.

## 4. Concurrency model

- **OCR throughput** scales horizontally: `docker compose up -d --scale worker=3`.
  Safe because `claim_next` uses `FOR UPDATE SKIP LOCKED`. Set workers ≈ available
  cores (CPU) or 1 (GPU).
- **Web concurrency:** the app is a single uvicorn process (sync endpoints run in a
  threadpool) — plenty for 3–4 users. If you want headroom, run `uvicorn … --workers
  2`. Note the app opens a short-lived Postgres connection per request (no pool); at
  this scale that's fine. If you ever scale web workers up, add a connection pool
  (`psycopg_pool`) — see §8.

## 5. Recommended hosting topology (single VM)

```
Internet ──► Caddy (TLS, :443, auto Let's Encrypt) ──► app:8000
                                                   (docker compose network)
             postgres + worker(s) stay internal; only 443 is public.
```

1. Provision a Linux VM (Ubuntu LTS), install Docker + Compose plugin.
2. Attach/format a persistent disk for `./data` and the `pgdata` volume; ensure it's
   included in disk snapshots.
3. Clone the repo, create `.env` from `.env.example` (see §6) with **strong**
   `AUTH_SECRET` + `ADMIN_PASSWORD` + `POSTGRES_PASSWORD`.
4. Put Caddy in front for TLS on a real domain (2-line `Caddyfile`:
   `your.domain { reverse_proxy localhost:8000 }`). **TLS is required** so the
   session cookie is safe (see §7).
5. `docker compose up -d --build` (optionally `--scale worker=N`).
6. First start seeds the admin from `.env`. Log in, change the admin password
   ideal-way (or rotate via env), create reviewers, assign ranges.

**Managed alternative (optional):** swap the `postgres` service for a managed
Postgres (RDS / Cloud SQL) and point `DATABASE_URL` at it — you get automated
backups + PITR for free. The app/worker containers are unchanged. This is the main
upgrade I'd suggest if the corpus is valuable, since **the DB is the irreplaceable
state**.

## 6. Configuration (env vars)

Set in `.env` (gitignored; never commit real secrets). Full template in
`.env.example`.

| Var | Purpose | Notes |
|---|---|---|
| `POSTGRES_USER/PASSWORD/DB` | DB credentials | compose builds `DATABASE_URL` from these |
| `DATABASE_URL` | Postgres DSN | set automatically in compose; override for managed PG |
| `AUTH_SECRET` | signs session JWTs | **long random** (`python -c "import secrets;print(secrets.token_urlsafe(48))"`) |
| `ADMIN_USERNAME` / `ADMIN_PASSWORD` | seeds first admin | only used when the users table is empty |
| `OCR_BACKEND` | `paddle` (default) / `vision` | |
| `OCR_USE_GPU` | `0` CPU (default) / `1` GPU | GPU needs a CUDA paddle wheel + NVIDIA runtime |
| `OCR_LANG` | `chinese_cht` (Traditional) | best for Sino-Nom |
| `PDF_DPI` | render DPI (default 300) | ↓ = faster OCR + smaller images, ↓ accuracy |
| `TRANSLATE_BACKEND` / `CORRECT_BACKEND` | `skip` (default) | server-side LLM is **off**; reviewers use their own keys in the UI |
| `WORKER_POLL_S` | queue poll interval (default 2s) | |

**No LLM API keys are needed on the server** — vision/correct/translate use each
reviewer's own key from the browser.

## 7. Security checklist

- [ ] **HTTPS only** via the reverse proxy (Let's Encrypt). Without it, the session
      cookie travels in clear.
- [ ] Set the session cookie `Secure` behind HTTPS (small code change — see §8; or
      terminate TLS and keep the app internal).
- [ ] Strong `AUTH_SECRET`, `POSTGRES_PASSWORD`, and a real `ADMIN_PASSWORD`
      (change from the dev default `admin123`).
- [ ] Postgres and workers are **not** exposed publicly — only the proxy's 443.
- [ ] Secrets live in `.env` / a secret manager, never in the image or git.
- [ ] Passwords are stored as **bcrypt hashes**; per-user LLM keys are never
      persisted or logged (verified in code).
- [ ] Restrict SSH / firewall to the ops team.

## 8. Reliability & ops runbook

**Backups (most important — the DB is the corpus):**
- Nightly `pg_dump`:
  `docker compose exec -T postgres pg_dump -U hannom hannom | gzip > backup_$(date +%F).sql.gz`
  Keep off-box (object storage). Or use managed Postgres for automated PITR.
- Also back up `data/uploads/` (source PDFs). `data/output/pages/` can be
  regenerated by re-running a job if lost.

**Restore:** `gunzip -c backup.sql.gz | docker compose exec -T postgres psql -U hannom hannom`.

**Scale OCR:** `docker compose up -d --scale worker=3` (add cores first).

**Deploy an update:** `git pull && docker compose up -d --build`. Schema migrations
are additive and applied automatically on startup; the queue drains safely.

**Logs / health:** `docker compose logs -f app worker`; `GET /healthz` for a
liveness probe; Postgres has a `pg_isready` healthcheck.

**Recommended enhancements for production (optional, small):**
1. **Set `Secure` on the session cookie** when served over HTTPS (add a
   `COOKIE_SECURE` env, pass `secure=True` in `set_cookie`). ~3 lines.
2. **Postgres connection pool** (`psycopg_pool`) if you run multiple uvicorn web
   workers — avoids per-request connect overhead. Not needed for 3–4 users on one
   web worker.
3. **Off-box backups + a snapshot schedule** on the data disk.

Items (1)–(2) are quick code changes I can make on request; nothing else needs to
change to host at this scale.

---

## 9. What the DevOps person needs to provision

- 1 Linux VM: **4 vCPU / 8 GB / 30 GB SSD** (or 8 vCPU / 16 GB for faster OCR; +GPU
  optional). Docker + Compose plugin.
- A **domain name** + reverse proxy (Caddy recommended) for TLS.
- A **persistent, snapshotted disk** for `./data` and `pgdata`.
- A **nightly Postgres backup** job (or a managed Postgres instance).
- Firewall: expose only 443; keep Postgres/worker internal.

That's the whole footprint. It's a small, self-contained system — one VM comfortably
serves 3–4 reviewers and a few-thousand-page corpus.
